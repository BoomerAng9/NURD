## 3. Google AI Backbone

NURDSCODE already points in the right cloud direction because GCP, Firebase, and Vertex AI are part of the stack. This add-on formalizes Google AI as a first-class backbone.

| Component | NURDSCODE role |
|---|---|
| Gemini Enterprise Platform | Enterprise-facing agent discovery, governance, and agent gallery/control plane |
| Gemini Agent Development Kit / ADK | Code-first agent creation and structured agent workflows |
| Vertex AI Agent Engine | Managed hosting target for ADK agents |
| Gemini CLI | Google-native terminal agent for repo tasks, coding, research, MCP, and shell workflows |
| Gemini Interactions API | Beta interface for stateful agent/model interactions, streaming, background agent tasks, and tool orchestration |
| Gemini Embeddings | Text and multimodal memory/search layer |
| Firebase | Auth, app hosting, storage, and realtime features where useful |
| GCP / Cloud Run | Owned service runtime, API services, and heavier task execution |
| Vertex AI | Enterprise-grade model access, agent deployment, grounding, governance, and privacy controls |

### Directive

```txt
Use Google AI Backbone as a native infrastructure and agent layer.
Do not make Gemini Enterprise the beginner customer surface.
Use Gemini Enterprise for enterprise agent governance and agent gallery needs.
Use ADK for Google-native agents.
Use Vertex AI Agent Engine for managed agent hosting where needed.
Use Gemini CLI for Google-native terminal work.
Use Gemini Embeddings for memory and retrieval.
Use Interactions API behind feature flags until stable for production use.
```

## 7. Gemini Interactions API directive

The Gemini Interactions API is powerful but must be handled carefully.

### NURDSCODE use cases

```txt
stateful builder conversations
agent task records
background Deep Research-style work
multimodal build planning
streaming builder feedback
tool-calling experiments
long-running research tasks
agent interaction receipts
```

### Restriction

```txt
Use Interactions API for beta agent lanes, internal experiments, background research, non-critical stateful workflows, and advanced builder experiences behind feature flags.

Do not use Interactions API as the default production inference path for paid customer builds until it stabilizes.

For stable production, use Vertex AI / Gemini APIs through approved production paths, use standard generateContent where applicable, and log all Interactions API use in the Evidence Layer.
```

### Privacy rule

```txt
For private paid builds:
- prefer Vertex AI production paths
- disable cache where zero-data-retention is required
- use store=false when using Interactions API and background/stateful behavior is not required
- do not use background=true for sensitive customer builds unless the retention model is disclosed and approved
```

```txt
semantic app memory

OpenKlass course/lab retrieval
image-to-app matching
voice-to-requirements matching
```

### Embedding policy

```txt
Use gemini-embedding-001 for stable production text/code retrieval.

Mark Gemini Embedding 2 as preview/public-preview until stable.
```

## 9. Privacy and paid-app training rule

NURDSCODE must make privacy part of the product surface.

### Product promise

```txt
Private Build Mode:
Paid customer projects are not used by NURDSCODE to train models.
Customer prompts, files, source code, generated apps, and project records stay tied to the customer workspace.
External AI providers are routed through Tool House-approved commercial/privacy-governed paths.
Customer data is not used to improve vendor models unless the customer explicitly opts into that provider's model-improvement program.
```

### Implementation rule

```txt
Default paid/private builds to:
- active billing / paid-service paths
- Vertex AI where enterprise controls are needed
- project-level cache controls where zero-data-retention is required
- Tool House provider data labels
- no customer-facing overclaims
```

## 21. Final directive

```txt
NURDSCODE must match VibeCode-level simplicity first, then exceed it through governed escalation.

The user starts with a private, low-friction idea capture surface. The platform must not overwhelm the user with infrastructure, models, agents, tools, or stack decisions.

The default product path is:
Idea -> Private Confirmation -> Simple Preview -> Guided Upgrade -> Evidence-Based Ship.

The advanced path is:
Idea or Repo -> A.I.M.S. Mission Anchor -> Spinner Route -> Tool House Selection -> OpenHands Onboarding if repo exists -> Claude/Gemini Agent Execution -> Warp/Oz Operator Review or Fan-Out if needed -> Self-Check Receipt -> Ship Gate Status -> Deployment/Bridge.

Google AI Backbone is now part of the NURDSCODE dev cycle:
- Gemini Enterprise Platform is the enterprise agent governance/control plane.
- Gemini ADK is the Google-native code-first agent framework.
- Vertex AI Agent Engine is the managed hosting lane for ADK agents.

- Gemini Interactions API is beta and must be feature-flagged.
- Gemini Embeddings provide semantic and multimodal retrieval.
- Firebase, GCP, Cloud Run, and Vertex AI remain the native infrastructure base.

OpenHands is the repo onboarding engine.
Warp is the product delivery workbench.
Claude Managed Agents are the precision coding/review layer.
Gemini ADK agents are the Google-native enterprise agent layer.
Spinner routes.
Stepper automates.
Tool House governs.
Self-Check proves.
```
